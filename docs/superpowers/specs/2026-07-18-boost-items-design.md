# Бусты из базы SC: предметы вместо абстрактных типов

**Дата:** 2026-07-18
**Статус:** одобрено пользователем (4 секции)
**Заменяет часть:** `2026-07-17-profile-boosts-design.md` (модель выбора и отображение)

## Суть

Игрок отмечает в профиле не абстрактные типы, а **конкретные предметы-усиления из базы STALCRAFT** (stalzone-database) — по одному предмету на каждый из четырёх типов: Усиление, Кратковременное усиление, Скорость, Защита. Показываются только предметы ранга **Ветеран и выше** (существующий фильтр `AllowedColors`). Тип предмета определяется автоматически из его JSON (блок «Назначение» / `effect_type`) — ручной разметки нет.

Маппинг `effect_type` → слот:

| effect_type (база SC) | Слот (BoostType) | Предметов Ветеран+ (на 2026-07-18) |
|---|---|---|
| `long_time_medicine` | Damage=0 «Усиление» | 32 |
| `short_time_medicine` | ShortDamage=1 «Кратк. усиление» | 9 |
| `mobility` | Speed=2 «Скорость» | 8 |
| `protection` | Defense=3 «Защита» | 3 |
| `healing`, `accumulation`, прочее/нет | — не буст, в слотах не ищется | — |

## 1. Синк и кэш предметов

`ItemSyncHostedService` (еженедельно, среда):

- Дополнительно к weapon/armor берёт из listing все `/items/supply/*` (~130 позиций).
- Для каждого supply-предмета скачивает его JSON (параллельно, батчем) и вычитывает `effect_type` из key-value блока с ключом `*.effect_type` (структура как в тултипе «Назначение»).
- `ItemDto` получает необязательное поле `BoostType?` — заполнено только у supply-предметов с бустовым effect_type. Существующие потребители (тикеты, инвентарь) не затронуты.
- Ошибка скачивания JSON одного предмета: предмет остаётся в кэше без типа буста, синк не падает (лог как сейчас).

## 2. БД и API

**БД.** `PlayerBoostRequest` + колонка `ItemId` (string, NOT NULL). Уникальный индекс `(UserId, BoostType)` уже обеспечивает «по одному на тип». Миграция: `DELETE FROM PlayerBoostRequests` (данные только тестовые), затем `ADD COLUMN ItemId`.

**PUT `api/profile/boosts`** (`[Authorize]`, UserId из токена) — тело: список пар `{boostType, itemId}`, 0–4 элемента.
Валидатор: тип в enum, itemId непустой, дубли типов запрещены.
Хендлер: через `IItemCacheService` проверяет, что itemId существует, его `BoostType` совпадает с переданным типом и ранг Ветеран+ — иначе `Result.Failure` → 400. Замена всего набора (как сейчас: remove+add, один SaveChanges).

**GET `api/profile/boosts`**, обогащение отрядов/профиля, **GET `api/boosts/summary`** (`[RankAuthorize(Member)]`) — отдают карточки `{boostType, itemId, name, icon}`; name/icon обогащаются из кэша на лету. Предмет исчез из кэша (патч игры): `name = itemId`, `icon = null` — не падаем. Агрегация «итого» считается на фронте, отдельного эндпоинта нет.

**GET `api/items/search`** + необязательный `boostType` — поиск только среди supply-бустов этого типа (существующие фильтры ранга и текста работают поверх). Наличие `boostType` снимает требование «минимум 2 символа или категория»: пустой `q` с `boostType` возвращает весь список типа (лимит 40, вариантов максимум 32).

## 3. Профиль: карточка «Нужные бусты»

Четыре слота (Усиление / Кратк. усиление / Скорость / Защита):

- Пустой слот — автокомплит-поиск (иконка + название в дропдауне), ищет только предметы своего типа. Дропдаун открывается по фокусу со всем списком (вариантов 3–32), без минимума символов.
- Заполненный слот — чип с иконкой, названием и ✕ для очистки; выбор нового предмета заменяет старый.
- Каждое изменение сразу PUT всего набора; optimistic-обновление, слоты disabled на время запроса (существующий паттерн).
- `BoostChips` переезжает на чипы предметов (иконка + имя) для read-only мест.

## 4. Read-only отображение и /boosts

- **Публичный профиль** — карточка «Нужные бусты»: чипы предметов с подписью типа мелким текстом; скрыта при пустом наборе.
- **Попап карточек отрядов** — компактные чипы (иконка + короткое название), до 4.
- **/boosts** (Участник+), два блока:
  - «Итого нужно» — по предметам: иконка, название, тип, «× N игроков»; сортировка N по убыванию.
  - «По игрокам» — ник (ссылка на профиль) → его чипы. Заменяет прежнюю таблицу и мобильные карточки; одна вёрстка на все экраны.
  - Пустое состояние — как сейчас.
- **i18n**: подписи слотов — существующие `boosts.types.*`; новые ключи: плейсхолдер поиска, заголовок «Итого нужно», «× N» (ru + en).

## Тесты

- Юнит: маппинг effect_type → BoostType в синке (включая «не буст» и отсутствие блока); валидация PUT (дубль типа, несуществующий предмет, чужой тип, ранг ниже Ветерана); обогащение карточек (включая исчезнувший из кэша предмет).
- Живая приёмка Playwright: заполнение слотов через поиск, замена и очистка, все три read-only поверхности, «Итого нужно», пустое состояние.
