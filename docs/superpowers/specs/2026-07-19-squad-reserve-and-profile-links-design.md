# Резерв во вкладке «Отряды» + клик по участнику → профиль — дизайн

Дата: 2026-07-19
Статус: утверждён

## Проблема

Во вкладке «Отряды» (`/squads`) не видно, кто из членов клана не состоит ни в одном отряде («в резерве») — этот список сейчас доступен только офицерам в билдере отрядов как «пул». Кроме того, нигде во вкладке «Отряды» нельзя кликнуть на участника, чтобы открыть его профиль — только карточка отряда целиком ведёт на страницу отряда, сами участники нигде не кликабельны.

## Решение

1. На `/squads` рядом с карточками отрядов появляется отдельная карточка «Резерв» со списком участников клана (Member+), которые не состоят ни в одном отряде. Видна всем Member+.
2. Везде, где во вкладке «Отряды» показывается ник участника (карточка отряда, страница отряда, карточка резерва), клик по нику ведёт на `/players/{userId}` — существующий маршрут просмотра чужого профиля (Member+, редиректит на `/profile`, если кликнули по себе). Права доступа не меняются.

## Бекенд

### Новый запрос: `GetSquadReserveQuery`

`Features/Squads/Queries/GetSquadReserve/`:
- `GetSquadReserveQuery : IRequest<IReadOnlyList<ReserveMemberDto>>` — без параметров.
- `ReserveMemberDto(Guid UserId, string Username, string? GameNickname, PlayerFlagsDto Flags, double? Kd, IReadOnlyList<BoostItemDto> Boosts)`.
- Хендлер: `userRepository.GetByMinRankAsync(UserRank.Member)` минус пользователи, входящие в `squadRepository.GetAllWithMembersAsync()` (по `UserId`, как в `GetSquadBuilderQueryHandler`); обогащение — `SquadMemberEnricher.ComputeAsync` (та же логика, что и в `GetSquadsQueryHandler`/билдере). Сортировка — по `Kd` по убыванию (null — в конец), как пул билдера.

### Эндпоинт

`GET /api/squads/reserve` в `SquadsController`, `[RankAuthorize(UserRank.Member)]` — рядом с существующим `GET /api/squads`. Отдельный маршрут, не меняющий форму ответа `GET /api/squads` (её уже потребляет дашборд как плоский `SquadDto[]`).

## Фронтенд

### API

`squadsApi.getReserve(): Promise<ReserveMemberDto[]>` → `GET /squads/reserve`. Тип `ReserveMemberDto` в `types/api.ts` (поля как в бекенд-DTO, camelCase).

### Карточка «Резерв» (`/squads`)

Новый компонент `components/squads/ReserveCard.tsx`, рендерится в `_auth.squads.index.tsx` в конце той же сетки карточек отрядов (`grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3`), отдельный `useQuery(['squads', 'reserve'], squadsApi.getReserve)`.
- Заголовок «Резерв» + счётчик участников.
- Список ников, каждый — `MemberHoverInfo` (как у участников отряда, тот же ховер-попап с флагами/КД/бустами) вокруг кликабельного `<Link to="/players/$userId">`.
- Без прогресс-бара заполненности и без ссылки-обёртки на всю карточку (резерв не ведёт никуда как единое целое).
- Пустое состояние: «Все бойцы в отрядах».
- Если резерв пуст, карточка всё равно рендерится (с пустым состоянием), чтобы не выглядело как баг загрузки.

### Клик по нику → профиль

Затрагиваются три места:

1. **`SquadCard.tsx`** (карточка отряда на `/squads`): сейчас вся карточка — один `<Link to="/squads/$squadId">`. Меняется на `<div onClick={...}>` с навигацией через `useNavigate()` на клик по карточке. Ники лидера и участников оборачиваются в `<Link to="/players/$userId" params={{userId}}>` с `onClick={(e) => e.stopPropagation()}` — клик по нику не всплывает до корневого `onClick` и не triggers переход на страницу отряда. Кнопка переименования (`startEdit`) и поле ввода уже вызывают `e.stopPropagation()` — их не трогаем. Hover-подсветка (`hover:text-accent`) добавляется на сам ник.
2. **`_auth.squads.$squadId.tsx`** (таблица участников отряда): ник в ячейке таблицы становится `<Link to="/players/$userId">`.
3. **`ReserveCard.tsx`**: ники сразу кликабельны (см. выше) — новый компонент, конфликтов с обёрткой нет.

Переименование отряда (`startEdit`) и кнопки офицеров (set-leader/remove) на странице отряда уже используют `stopPropagation`/не находятся внутри кликабельной карточки — не затрагиваются.

## Крайние случаи

- Участник в резерве без `gameNickname` — показывается `username`, как и в существующих списках.
- Клик по своему нику (свой отряд, свой резерв) — маршрут `/players/{userId}` уже редиректит на `/profile`, если `userId === user.userId`; отдельно обрабатывать не нужно.
- Гость (rank < Member) не видит вкладку «Отряды» вовсе (текущее поведение не меняется) — вопрос видимости резерва гостям не возникает.

## Тестирование

- Backend: юнит-тесты `GetSquadReserveQueryHandlerTests` по образцу `GetSquadsQueryHandlerTests` — участник в отряде исключён из резерва; участник без отряда, но с рангом < Member исключён; сортировка по КД; обогащение флагами/бустами через мок `SquadMemberEnricher`-зависимостей.
- Frontend: ручная проверка на дев-стенде — резерв виден и пуст/непуст корректно; клик по участнику отряда открывает его профиль, а клик по остальной части карточки — страницу отряда; клик по участнику резерва открывает профиль.
