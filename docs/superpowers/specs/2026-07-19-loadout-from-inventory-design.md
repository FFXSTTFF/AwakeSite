# Смена экипировки из инвентаря — дизайн

Дата: 2026-07-19
Статус: утверждён

## Проблема

Экипировка в профиле игрока (снайперка / основное оружие / броня + заточка) берётся из последней заявки с заполненным `/loadout` и «заморожена» с момента подачи заявки. Игрок не может её обновить. Инвентарь при этом уже хранит актуальный список брони и оружия игрока.

## Решение

Игрок редактирует карточку «Экипировка» в своём профиле, выбирая предметы **только из своего инвентаря**. Заточка (0–15) указывается вручную при надевании. Надетая экипировка хранится персонально у пользователя.

## Бекенд

### Domain

- `User` получает свойство `public Loadout? Loadout { get; set; }` — существующий value object `Awake.Domain.ValueObjects.Loadout` (`Sniper?`, `Weapon`, `Armor`; каждый слот — `LoadoutSlot(ItemId, ItemName, ItemIcon, Upgrade)`).

### Infrastructure

- В `UserConfiguration` — jsonb-конвертация свойства `Loadout`, копия паттерна из `TicketConfiguration` (camelCase `JsonSerializerOptions`, `HasColumnType("jsonb")`).
- Миграция `AddUserLoadout` (nullable jsonb-колонка `Loadout` в таблице пользователей).

### Application

- Команда `UpdateMyLoadoutCommand(Guid UserId, LoadoutSlotRequest? Sniper, LoadoutSlotRequest? Weapon, LoadoutSlotRequest? Armor)` → `Result<bool>` в `Features/Inventory/Commands/UpdateMyLoadout/`, где `LoadoutSlotRequest(string ItemId, int Upgrade)` — клиент шлёт только id предмета и заточку.
- Валидация в хендлере:
  - оружие и броня обязательны (снайперка опциональна);
  - заточка каждого слота в диапазоне 0–15;
  - `ItemId` каждого заполненного слота присутствует в инвентаре игрока (`IPlayerInventoryRepository`);
  - предмет есть в кэше базы предметов (`IItemCacheService`) — имя и иконку слота сервер резолвит сам, не доверяя клиенту;
  - нарушение → `Result.Failure` с русскоязычным сообщением.
- Хендлер собирает `Loadout` из отрезолвленных слотов, записывает `user.Loadout` и сохраняет.

### API

- `PUT /api/profile/loadout` в `InventoryController` (там уже живут `/api/profile/inventory` и `/api/profile/build-proof`). Тело — `{ sniper: {itemId, upgrade} | null, weapon: {itemId, upgrade}, armor: {itemId, upgrade} }` (camelCase JSON). Авторизованный пользователь меняет только свою экипировку.

### Чтение профиля

- `GetPlayerProfileQueryHandler`: `loadout = user.Loadout ?? (экипировка из последней заявки, как сейчас)`. Существующие игроки без сохранённой экипировки ничего не теряют.

## Фронтенд

### API-клиент

- `inventoryApi.updateLoadout(loadout: Loadout): Promise<void>` → `PUT /profile/loadout`.

### Карточка «Экипировка» (`PlayerProfileView`)

- Новый проп `editable` (true только на странице своего профиля `_auth.profile.tsx`).
- Для своего профиля карточка видна всегда, даже при `loadout === null` (сейчас скрыта) — с подсказкой «выбери из инвентаря».
- Кнопка «Изменить» → режим редактирования:
  - три слота: снайперка (опционально, можно убрать), основное оружие, броня;
  - выбор в каждом слоте — выпадающий список из предметов инвентаря, отфильтрованных по категории:
    - броня → `category` начинается с `armor/`;
    - снайперка → `category === "weapon/sniper_rifle"`;
    - основное оружие → остальные `weapon/*`;
    - предметы с `unknown === true` исключаются;
  - у каждого заполненного слота поле заточки 0–15;
  - если в инвентаре нет предметов нужной категории — подсказка «добавь предмет в инвентарь»;
  - «Сохранить» → мутация + инвалидация запроса профиля; «Отмена» → выход без изменений.
- Чужие профили — только просмотр, без изменений.

## Крайние случаи

- Удаление предмета из инвентаря **не** сбрасывает надетую экипировку: она — снапшот (имя + иконка + заточка), как в заявке. Предмет просто исчезает из будущего выбора.
- Валидация «предмет в инвентаре» выполняется только при сохранении экипировки.
- Пока игрок не сохранил свою экипировку, профиль показывает экипировку из заявки (fallback).

## Тестирование

- Юнит-тесты `UpdateMyLoadoutCommandHandler` по образцу существующих тестов команд инвентаря:
  - успешное сохранение;
  - предмет отсутствует в инвентаре → Failure;
  - заточка вне диапазона → Failure;
  - пользователь не найден → Failure.
- Ручная сквозная проверка на локальном стенде (vite 5173 → api 5001).
